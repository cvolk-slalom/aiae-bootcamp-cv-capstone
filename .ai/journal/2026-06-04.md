# 2026-06-04 — F005 — Bed layout (greedy packer + editable grid)

## 02:00 — F005 done

**Did:**
- `shared`: extended [PlanLayoutSchema](../../packages/shared/src/plan.ts) with `cols`/`rows`; added `LayoutResultSchema` + `UpdateLayoutRequestSchema` aliases in [api.ts](../../packages/shared/src/api.ts).
- `api`:
  - Pure service [services/layout.ts](../../apps/api/src/services/layout.ts): `computeLayout(input, db)` and `validateLayout(layout, bed, confirmed)`. Cell size = max-spacing of confirmed plants clamped 6–24 in; `cols/rows = floor(bed / cell)`; greedy packer sorts by spacing desc, each plant claims a `ceil(spacing/cell)²` footprint in row-major order, overflow → `unplaced`; if `lightHours < 6`, full-sun plants demote to `unplaced` before packing.
  - Routes ([routes/plans.ts](../../apps/api/src/routes/plans.ts)): `GET /plans/:id/layout/suggestion` (non-persisting) and `PATCH /plans/:id/layout` (Zod + structural validation: dims match `inputs.bed`, no overlapping cells, every `plantId` ∈ `confirmedPlants ∪ {null}`; persists, advances `step → 'timing'`).
  - Tests: [services/layout.test.ts](../../apps/api/src/services/layout.test.ts) (7: empty / 6-plant pack / oversized → unplaced / low-light demote / 3 validateLayout); 4 new flow tests in [routes/plans.test.ts](../../apps/api/src/routes/plans.test.ts) (suggestion shape / patch+reload+step / unknown id 400 / dimension mismatch 400).
- `web`:
  - API client ([api/client.ts](../../apps/web/src/api/client.ts)): `getLayoutSuggestion`, `updateLayout`.
  - [Layout.tsx](../../apps/web/src/screens/Layout/Layout.tsx) — loads `plan.layout` if present else the suggestion; CSS-grid table with a per-cell `<select>` of confirmed plants (or empty); unplaced list; "Regenerate suggestion" + "Save & continue" → `/plans/:id/timing`. Added `/plans/:id/timing` placeholder in [App.tsx](../../apps/web/src/App.tsx).

**Verified:**
- `pnpm test` → **26 passing** (1 shared, 12 plans, 7 layout, 6 companions, 1 server).
- `pnpm lint`, `pnpm build` → 0 errors; web bundle 232 KB / 70 KB gz.
- Live `tsx` server: 48×96 bed + 6 confirmed → `cellSizeIn: 24`, `2×4` grid, all 6 plants placed (`unplaced: []`), PATCH advanced `step` to `timing`. Low-light (`lightHours: 4`) with `[tomato, basil]` → both in `unplaced` (both full sun). Dimension-tampered PATCH returned 400.

**Why:** F005 needed deterministic, simple packing that the brief could fully specify and a user could override in one click. Pure-function service keeps the recommender + layout in the same pattern: route is plumbing, service is logic, dataset is data.

**AI leverage:**
- **Same recipe two features in a row.** F004 established `services/<x>.ts` (pure) + Zod result schema in `shared` + route plumbing + a screen that loads/edits/saves. F005 cloned the recipe with zero new conventions to invent — the brief enumerated acceptance, the build was a checklist walk.
- **Validation done twice, deliberately.** Zod (`UpdateLayoutRequestSchema = PlanLayoutSchema`) handles shape; a separate `validateLayout()` handles cross-field invariants (dims-match-bed, unknown plantIds, cell uniqueness). Split keeps each layer small and testable in isolation, and the structural errors surface in the same `{formErrors, fieldErrors}` envelope every other route uses → frontend renders them with zero new code.
- **Greedy + 1-cell-min footprint is a one-screen reading.** Documented the footprint formula inline (`ceil(spacingIn/cellSizeIn)²`); the entire service is ~70 lines. Future smarter ranking is a new brief + new ADR — explicitly *not* this feature.
- **E2e smoke caught a wrong assumption fast.** First curl attempt used `node dist/server.js` after `pnpm build` — but `build` is `tsc --noEmit` (no compile output). Switched to `pnpm --filter @gpb/api start` (tsx). 30-second discovery, zero changes to actual code.
- **Tests-as-acceptance, again.** [layout.test.ts](../../apps/api/src/services/layout.test.ts) reads top-to-bottom as the brief: empty list, oversized-plant case, low-light case. Anyone auditing F005 reads that file and is done.

**Patterns / learnings:**
- **`PlanLayoutSchema` got `cols`/`rows` added** — they're derivable from `cellSizeIn` + bed but persisting them simplifies the validator and the frontend (no recompute on load). When a derived value is read 10× per render and written once, store it.
- **Light filter runs *before* cell-size selection.** Otherwise a full-sun plant with the largest spacing would inflate `cellSizeIn` even though the plant ends up in `unplaced`. Order matters; documented by the test that pins `cellSizeIn` for the low-light case implicitly.
- **Reused `PlantsDb` type from companions service.** No new abstraction, no new import path — the existing `Map<string, Plant>` flows through both services and the route registrations.

**Decisions:** none new (still bound by ADR 0003; layout is intentionally simple per the brief).

## 02:10 — F006 — Frost & window timing

**Did:**
- `shared` ([api.ts](../../packages/shared/src/api.ts)): `TimingResultSchema` (alias of `PlanTimingSchema`) and a stricter `UpdateTimingRequestSchema` that requires ISO `YYYY-MM-DD` dates (or omitted). `PlanTimingSchema` itself stays permissive for storage compatibility.
- `api`:
  - Pure service [services/timing.ts](../../apps/api/src/services/timing.ts): `addDays` / `addWeeks` (UTC ms math, no timezone), `computeTiming(lastFrostDate, confirmedPlants, db)`. Per plant: `startIndoorsOn = lastFrost - n weeks`, `transplantOn = lastFrost + n weeks`, `directSowOn = lastFrost ± n weeks`. Harvest derives from `transplantOn ?? directSowOn` + `daysToMaturity` days; `harvestEnd = harvestStart + 21 days`. Plants with no timing data and unknown ids yield `{ plantId }`-only rows.
  - Routes ([routes/plans.ts](../../apps/api/src/routes/plans.ts)): `GET /plans/:id/timing/suggestion` (non-persisting) and `PATCH /plans/:id/timing` (Zod ISO check + every `plantId ∈ confirmedPlants`; persists; advances `step → 'final'`).
  - Tests: [services/timing.test.ts](../../apps/api/src/services/timing.test.ts) (6: `addWeeks` ±, brief acceptance for tomato @ 2026-05-01, direct-sow case, no-timing-data, unknown id); 4 new flow tests in [routes/plans.test.ts](../../apps/api/src/routes/plans.test.ts) (suggestion matches brief / patch advances to final + reload / non-ISO 400 / unknown id 400).
- `web`:
  - API client ([api/client.ts](../../apps/web/src/api/client.ts)): `getTimingSuggestion`, `updateTiming`.
  - [Timing.tsx](../../apps/web/src/screens/Timing/Timing.tsx) — table with one row per confirmed plant, five editable `<input type="date">` cells (start indoors / direct sow / transplant / harvest start / harvest end). Loads `plan.timing` if present else the suggestion. "Save & continue" → `/plans/:id/final` (placeholder added in [App.tsx](../../apps/web/src/App.tsx)).

**Verified:**
- `pnpm test` → **36 passing** (1 shared, 16 plans, 7 layout, 6 timing, 6 companions). All four new timing service tests pass; `addWeeks('2026-05-01', -6) === '2026-03-20'`.
- `pnpm lint`, `pnpm build` → 0 errors; web bundle 236 KB / 71 KB gz.
- Live `tsx` server: created plan w/ `lastFrostDate: 2026-05-01`, walked inputs→companions→layout→timing. Tomato suggestion: `startIndoorsOn: 2026-03-20`, `transplantOn: 2026-05-15`, `harvestStart: 2026-07-29`, `harvestEnd: 2026-08-19` — exact match to brief acceptance. PATCH advanced `step` to `final`. `{"startIndoorsOn":"nope"}` → 400.

**Why:** F006 turns the user's `lastFrostDate` into actionable per-plant dates — the first time the wizard outputs *answers* rather than *catalogues*. Keeping it as pure UTC math means tests are deterministic across machines/timezones and there's no Date-library footgun to maintain.

**AI leverage:**
- **Brief acceptance = the test name.** The first `computeTiming` test reads "matches the brief acceptance for tomato @ 2026-05-01" and asserts the four exact dates from the brief. Anyone auditing F006 reads that one test and is done — the brief and the test are the same artifact.
- **Recipe held for the third consecutive feature.** F004 (companions), F005 (layout), now F006 (timing) all follow the same shape: `services/<x>.ts` (pure, ~70 lines) → schemas in `shared` → two routes in `plans.ts` (`GET .../suggestion` non-persisting + `PATCH ...` persisting + advancing `step`) → one screen that loads-or-suggests, edits, saves, redirects. Three features, zero new conventions, ~25 mins of building each on top of stable scaffolding.
- **Schema split: storage permissive, request strict.** `PlanTimingSchema` keeps dates as `z.string().optional()` (so existing rows always round-trip), but `UpdateTimingRequestSchema` requires `YYYY-MM-DD`. The boundary is exactly where it needs to be — user input — and old data can't fail to load.
- **Pure UTC date math, no dependencies.** Refused the impulse to pull in `date-fns` or `dayjs` for two helper functions. `Date.UTC(...)` + ms arithmetic + `.toISOString().slice(0,10)` covers every case in the brief, including the negative-week traversal across a month boundary (2026-05-01 → 2026-03-20). Smaller bundle, fewer transitive deps, fully testable.
- **`<input type="date">` does the validation work for the user.** Browser-native datepicker emits ISO strings, which the request schema accepts as-is. Empty string deletes the field via `delete next[field]`. No custom datepicker, no parsing — the platform handles the round trip.

**Patterns / learnings:**
- **Stale dev server bit me during e2e.** A previous `pnpm dev` session had left a tsx process bound to `:3001` with the *old* route table; my new background process exited with `SQLITE_IOERR_SHORT_READ` (db file half-flushed), and `curl` happily hit the stale one returning 404 for `/timing/suggestion`. Fix: `pkill -f 'tsx src/server.ts'` *before* starting a fresh one and verify the new pid is alive (`ps -p $APID`) before issuing requests. Future e2e scripts should always do both.
- **`build = tsc --noEmit` in this repo.** Confirmed (again) that `node dist/server.js` doesn't exist. The right e2e binary is `pnpm --filter @gpb/api start` (tsx). Could codify a `start:js` if/when we add a real compile step, but not now — premature.
- **Per-feature delta is shrinking.** F003 took the most code (full CRUD scaffold); F004/F005/F006 each ship in a tighter diff because the surrounding infrastructure (Zod-shared, request envelope, error UI, screen layout, route shape) is reused untouched. The scaffolding compounds — exactly the AI-SDLC bet.
