# 2026-06-04 — F005 — Bed layout (greedy packer + editable grid)

## 02:00 — F005 done

**Did:**

- `api`:
  - Pure service [services/layout.ts](../../apps/api/src/services/layout.ts): `computeLayout(input, db)` and `validateLayout(layout, bed, confirmed)`. Cell size = max-spacing of confirmed plants clamped 6–24 in; `cols/rows = floor(bed / cell)`; greedy packer sorts by spacing desc, each plant claims a `ceil(spacing/cell)²` footprint in row-major order, overflow → `unplaced`; if `lightHours < 6`, full-sun plants demote to `unplaced` before packing.

**Why:** F005 needed deterministic, simple packing that the brief could fully specify and a user could override in one click. Pure-function service keeps the recommender + layout in the same pattern: route is plumbing, service is logic, dataset is data.

**AI leverage:**
- **Same recipe two features in a row.** F004 established `services/<x>.ts` (pure) + Zod result schema in `shared` + route plumbing + a screen that loads/edits/saves. F005 cloned the recipe with zero new conventions to invent — the brief enumerated acceptance, the build was a checklist walk.
- **Validation done twice, deliberately.** Zod (`UpdateLayoutRequestSchema = PlanLayoutSchema`) handles shape; a separate `validateLayout()` handles cross-field invariants (dims-match-bed, unknown plantIds, cell uniqueness). Split keeps each layer small and testable in isolation, and the structural errors surface in the same `{formErrors, fieldErrors}` envelope every other route uses → frontend renders them with zero new code.
- **Greedy + 1-cell-min footprint is a one-screen reading.** Documented the footprint formula inline (`ceil(spacingIn/cellSizeIn)²`); the entire service is ~70 lines. Future smarter ranking is a new brief + new ADR — explicitly *not* this feature.
- **E2e smoke caught a wrong assumption fast.** First curl attempt used `node dist/server.js` after `pnpm build` — but `build` is `tsc --noEmit` (no compile output). Switched to `pnpm --filter @gpb/api start` (tsx). 30-second discovery, zero changes to actual code.
- **Tests-as-acceptance, again.** [layout.test.ts](../../apps/api/src/services/layout.test.ts) reads top-to-bottom as the brief: empty list, oversized-plant case, low-light case. Anyone auditing F005 reads that file and is done.

**Patterns / learnings:**
- **`PlanLayoutSchema` got `cols`/`rows` added** — they're derivable from `cellSizeIn` + bed but persisting them simplifies the validator and the frontend (no recompute on load). When a derived value is read 10× per render and written once, store it.
- **Light filter runs *before* cell-size selection.** Otherwise a full-sun plant with the largest spacing would inflate `cellSizeIn` even though the plant ends up in `unplaced`. Order matters; documented by the test that pins `cellSizeIn` for the low-light case implicitly.
- **Reused `PlantsDb` type from companions service.** No new abstraction, no new import path — the existing `Map<string, Plant>` flows through both services and the route registrations.

**Decisions:** none new (still bound by ADR 0003; layout is intentionally simple per the brief).
