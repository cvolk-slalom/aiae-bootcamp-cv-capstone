- **`useMemo` over `useEffect` for derived form validity.** No state for "is valid" — just a memo that runs Zod on the form and returns the parsed value or null. Drives the submit button's `disabled`.

**Verified:**
- `pnpm test` → 16 passing across 4 files (8 plans + 6 companions + 1 server + 1 shared).
- `pnpm lint`, `pnpm build` → 0 errors; web bundle 229 KB / 69 KB gz.
- Live curl: `[tomato, basil, fennel]` → recommends `[carrot, onion, marigold, nasturtium, garlic]` (none are antagonists of any desired) and discourages `{a: fennel, b: tomato, reason: "Fennel and Tomato are antagonists"}`. PATCH advanced `step` to `layout` and confirmed list persisted across reload. Unknown id returned 400 with `formErrors: ["unknown plant id(s): not-a-plant"]`. `/`, `/plans/:id/companions`, `/plans/:id/layout` all 200.

**Why:** F004 is the first place the app does *thinking* on top of stored data. Keeping the recommender as a **pure function** of `(desired, plantsDb)` means it's trivially testable and trivially replaceable — when smarter ranking lands later, only `services/companions.ts` changes and the route stays the same.

**AI leverage:**
- **Pure-function service + JSON dataset = property-style tests for free.** The same `plants.json` loaded by the live server is loaded by the unit tests, so the test "no recommendation is an antagonist of any desired" is checked against real data, not a fixture.
- **Same Zod schema validates the route response and the client parse.** `CompanionsResultSchema` flows through `request<T>` in [client.ts](../../apps/web/src/api/client.ts) so a backend shape drift fails the frontend at parse time, in dev, instantly.
- **GET-rec is intentionally non-persisting.** The brief said so, and it pays off: recomputing on each load means edits to the underlying `plants.json` (or future recommender logic) show up the next time the user opens the screen — no migration, no stored-recommendation drift.
- **Dataset edit as part of the feature.** Adding `fennel` to satisfy a brief's acceptance case is cheaper than rewriting the brief to match a thin dataset. Single-file, single-line addition, plant.test.ts re-validates id integrity automatically.

**Patterns / learnings:**
- **Symmetric antagonism via lex-dedup.** Both tomato→fennel and fennel→tomato exist in the dataset (cleaner than picking a "primary" direction), so the service emits each pair once by sorting `(a, b)`. Avoids a duplicate-flag bug without forcing the dataset to be asymmetric.
- **`exactOptionalPropertyTypes`-style strict TS** caught `pairs[0].reason` (possibly undefined after `toHaveLength(1)`). Used `pairs[0]?.reason` rather than non-null `!` — minor but keeps the code honest if the test ever changes shape.
- **Web feature replaces, not augments.** Deleted the F003 `CompanionsPlaceholder` rather than guarding behind a flag. Per coding-standards: no feature flags when you can change code directly.

**Decisions:** none new (recommender stays simple per the brief; if smarter ranking is needed later, that's a new brief + ADR).
